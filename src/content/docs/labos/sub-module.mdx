Un **sous-module** représente une **extension** fonctionnelle ou technique qui vient compléter un module existant.
Autrement dit, il s’agit d’un module dépendant d’un autre, destiné à enrichir, modifier ou spécialiser 
ses fonctionnalités sans altérer directement son code source. 
Cette approche favorise la **modularité**, la réutilisation et la maintenabilité du code, 
des principes essentiels dans le développement logiciel professionnel.

Un sous-module est souvent utilisé pour :

- Étendre un module standard d’Odoo (par exemple, ajouter un champ ou une vue spécifique à un module de facturation);
- Adapter une fonctionnalité à un contexte particulier (comme une localisation comptable propre à un pays);
- Séparer des développements spécifiques afin de préserver la stabilité du module principal.

Sur le plan technique, le développement d’un sous-module suit la même structure qu’un module classique : 
- un dossier contenant un fichier de manifeste (__manifest__.py);
- des modèles Python;
- des vues XML;
- des règles d’accès.

La différence majeure réside dans la déclaration des dépendances : 
le sous-module indique explicitement **le module parent** dans la clé depends de son **manifeste**. 
Ainsi, Odoo charge d’abord le module parent, puis le sous-module, ce qui permet à ce dernier 
de surcharger ou d’étendre les éléments existants (modèles, vues, actions, menus).

## Le module `todo_stage`

Nous souhaitons **étendre** le module de gestion des tâches avec de nouvelles fonctionnalités. 
Pour ce faire, vous allez créer un **sous-module** intitulé `todo_stage` qui va permettre d'étendre 
le module de base **sans modifier** son code. Afin de démarrer cette extension :

1. **Créez** un nouveau module via la commande `scaffold`

2. **Mettez à jour le manifest** (`__manifest__.py`) de ce sous-module et **précisez** sa dépendance avec le module `todo_app` via l’option `depends` :
    ```python
    'depends': ['todo_app'],
    ```
    Pensez-vous qu'il soit normale que la dépendance au module `base` ait disparu ?
